A* Algorithm for Multi-Task Delivery Route Optimization.

This module implements an A* search algorithm adapted for the Vehicle Routing Problem (VRP) with capacity and fuel constraints. The algorithm finds the optimal sequence of pickups and deliveries that minimizes the total travel time.

The problem is modeled as a search tree where:
  • Each node represents a state (current location + completed tasks)
  • Edges represent transitions (move to pickup or delivery location)
  • Heuristic h(n) estimates the remaining cost to complete all tasks
  • Cost g(n) is the accumulated time since the start
Key Features:
  • Dijkstra cache to avoid recalculation of routes
  • Admissible heuristic based on average cost per task
  • Active task penalization (λ-penalty) to avoid overload
  • Capacity and fuel constraints verified dynamically
  • Search tree visualization with matplotlib/networkx

This algorithm does not follow the FIPA protocol as it is a computational optimization component that operates internally without agent communication. However, it integrates with the SPADE multi-agent system where agents may use this algorithm to plan routes and communicate the results via FIPA-ACL messages.

Classes:

TreeNode: Represents a state in the A* search tree.

Functions:

get_dijkstra_cached: Returns Dijkstra result with caching. clear_dijkstra_cache: Clears the global route cache. calculate_heuristic: Calculates h(n) for a given state. A_star_task_algorithm: Executes A* and returns the optimal route.

>>> path, time, tree = A_star_task_algorithm(
...     graph=graph,
...     start=0,
...     tasks=orders,
...     capacity=50,

>>> print(path)  # [(0, None), (3, 1), (5, 1), (7, 2), ...]
>>> print(f"Total time: {time}s")
>>> tree.plot_tree("search_tree.png")  # Optional visualization
Technical Notes:
  • Optimizations: Dijkstra cache reduces calls to routing algorithm
  • Admissibility: h(n) is admissible if average_cost ≤ actual minimum cost
  • Complexity: O(b^d) where b is branching factor and d is depth (2*num_tasks)
  • Memory: Stores entire search tree for visualization (can be disabled)
Algorithm Workflow:
  1. Initialize root node at starting location with all tasks pending
  2. Compute average cost per task for heuristic estimation
  3. Expand nodes with lowest f(n) = g(n) + h(n) using priority queue
  4. For each node, evaluate available pickup/delivery points considering:
    • Capacity constraints (current_load + task_quantity ≤ max_capacity)
    • Fuel constraints (route_fuel ≤ max_fuel)
    • Task precedence (pickup must occur before delivery)
  5. Generate child nodes for each feasible transition
  6. Continue until reaching target depth (2 * num_tasks)
  7. Backtrack from goal node to reconstruct optimal path
Performance Considerations:
  • Dijkstra cache reduces computational overhead significantly
  • Heuristic quality impacts search efficiency (better h → fewer expansions)
  • Lambda penalty parameter affects route structure (higher λ → fewer concurrent tasks)

 89def get_dijkstra_cached(graph: Graph, start: int, end: int):
 90    """Returns Dijkstra algorithm result using cache to avoid recalculations.
 91    
 92    Implements a memoization system for Dijkstra algorithm calls. Since the graph
 93    does not change during A* execution, routes between the same points will always
 94    yield the same result. The cache is indexed by (start, end) tuple.
 95    
 96    This function is a pure optimization technique and does not involve FIPA protocol.
 97    It operates at the computational level within a single agent's decision-making process.
 98    
 99    Args:
100        graph (Graph): Graph instance containing the network topology.
101        start (int): Source node ID.
102        end (int): Destination node ID.
103    
104    Returns:
105        tuple: A tuple (path, fuel, time) where:
106            - path (list[Node]): List of nodes in the shortest path.
107            - fuel (float): Fuel required to traverse the path.
108            - time (float): Total travel time in seconds.
109    
110    Side Effects:
111        Modifies the global _dijkstra_cache dictionary by adding new results.
112    
113    Examples:
114        >>> result1 = get_dijkstra_cached(graph, 3, 7)  # Computes and stores
115        >>> result2 = get_dijkstra_cached(graph, 3, 7)  # Returns from cache (instantaneous)
116        >>> assert result1 == result2
117        >>> # Cache hit provides O(1) lookup vs O(V²) for full Dijkstra
118    
119    Notes:
120        - Cache is global and persists between A* calls
121        - Use clear_dijkstra_cache() to clear before new execution
122        - Performance: O(1) for cache hits, O(V²) for misses (full Dijkstra)
123        - Memory: O(N²) worst case if all node pairs are queried
124    
125    Implementation Details:
126        The cache uses a dictionary with (start, end) tuples as keys.
127        Cache misses trigger a call to graph.djikstra() which implements
128        Dijkstra's algorithm with priority queue (typically O(E log V) complexity).
129    """
130    cache_key = (start, end)
131    if cache_key not in _dijkstra_cache:
132        _dijkstra_cache[cache_key] = graph.djikstra(start, end)
133    return _dijkstra_cache[cache_key]

311    def __gt__(self, other):
312        """Comparison operator greater than (>) based on f(n).
313        
314        Used by PriorityQueue to order nodes. Nodes with lower f(n) have
315        higher priority (are expanded first) in the A* algorithm.
316        
317        This method enables the priority queue to maintain nodes sorted by
318        their evaluation function, which is crucial for A* optimality.
319        
320        Args:
321            other (TreeNode): Another TreeNode to compare against.
322        
323        Returns:
324            bool: True if self.f > other.f, False otherwise.
325        
326        Notes:
327            - Lower f(n) means better (more promising) node
328            - PriorityQueue uses min-heap, so __gt__ reverses order
329            - Ties in f(n) are broken by object ID in PriorityQueue
330        
331        Examples:
332            >>> node1 = TreeNode(..., g=10, h=5)  # f = 15
333            >>> node2 = TreeNode(..., g=8, h=4)   # f = 12
334            >>> node1 > node2  # True (15 > 12)
335            >>> # In PriorityQueue, node2 will be expanded first
336        """
337        return self.f > other.f

388    def create_childs(self):
389        """Expands current node by creating all viable child nodes.
390        
391        For each available point (pickup or delivery), creates a new child node
392        representing the transition to that point. Updates state, load, costs,
393        and calculates new heuristic.
394        
395        This method implements the core expansion logic of the A* search tree.
396        Each child represents a decision to visit a specific pickup or delivery
397        location next in the route.
398        
399        Algorithm:
400            1. For each point in available_points:
401               - Copy lists of reached points
402               - Update load (+ for pickup, - for delivery)
403               - Create new TreeNode with depth+1
404               - Calculate g(n) = parent.g + travel_time
405               - Calculate h(n) with new state
406               - Add child to children list
407        
408        Side Effects:
409            - Modifies self.children (adds new nodes)
410            - Each child has reference to self as parent
411        
412        available_points Format:
413            List of tuples (location, orderid, quantity, time, type) where:
414            - location (int): Destination node ID
415            - orderid (int): Order ID
416            - quantity (int): Quantity to load/unload
417            - time (float): Travel time to location
418            - type (int): 1=warehouse (pickup), 0=customer (delivery)
419        
420        Examples:
421            >>> node.available_points = [(3, 1, 10, 5.5, 1), (7, 2, 15, 8.2, 1)]
422            >>> node.create_childs()
423            >>> print(len(node.children))  # 2
424            >>> print(node.children[0].location)  # 3
425            >>> print(node.children[0].quantity)  # 10 (pickup)
426        
427        Notes:
428            - Must call evaluate_available_points() before this method
429            - Children inherit state, max_quantity, max_fuel from parent
430            - order_id associated with child identifies which order generated transition
431            - Each child's g(n) includes cumulative cost from root
432            - Each child's h(n) is recalculated based on remaining tasks
433        
434        Implementation Details:
435            The method preserves immutability of parent state by copying lists.
436            Quantity is updated based on point type (warehouse adds, customer subtracts).
437            The heuristic calculation considers the new completion state.
438        """

452                location=point[0],
453                state=self.state,
454                max_quantity=self.max_quantity,
455                max_fuel=self.max_fuel,

457                depth=self.depth + 1,

475    def evaluate_available_points(self,graph: Graph):
476        """Evaluates which points (pickups/deliveries) are reachable from current node.
477        
478        Checks all tasks in state and determines which transitions are viable
479        considering:
480            - Capacity constraints (not exceeding max_quantity)
481            - Fuel constraints (not exceeding max_fuel)
482            - Logical sequence (pickup before delivery)
483            - Already visited points (avoid duplicates)
484        
485        This method implements the constraint checking logic that ensures generated
486        routes are feasible with respect to vehicle limitations.
487        
488        Decision Logic:
489            For each task (sender, receiver, qty, orderid) in state:
490            
491            1. If pickup not yet performed:
492               - Check if qty + current_load ≤ max_quantity
493               - Calculate route and fuel using get_dijkstra_cached
494               - If fuel ≤ max_fuel: add sender to available points (type=1)
495            
496            2. If pickup already performed but delivery not:
497               - Calculate route and fuel to receiver
498               - If fuel ≤ max_fuel: add receiver to available points (type=0)
499        
500        Args:
501            graph (Graph): Graph instance to calculate routes via Dijkstra.
502        
503        Returns:
504            list[tuple]: List of tuples (location, orderid, quantity, time, type) where:
505                - location (int): Destination node ID
506                - orderid (int): Associated order ID
507                - quantity (int): Quantity to load/unload
508                - time (float): Travel time to location in seconds
509                - type (int): 1=warehouse (pickup), 0=customer (delivery)
510        
511        Examples:
512            >>> node = TreeNode(location=0, state=[(3, 7, 10, 1), (5, 9, 15, 2)], ...)
513            >>> available = node.evaluate_available_points(graph)
514            >>> print(available)
515            # [(3, 1, 10, 5.5, 1), (5, 2, 15, 8.2, 1)]  # Two pickups available
516            
517            >>> # After picking up order 1:
518            >>> node2 = TreeNode(location=3, initial_points_reached=[(1, 3)], ...)
519            >>> available2 = node2.evaluate_available_points(graph)
520            >>> print(available2)
521            # [(7, 1, 10, 3.2, 0), (5, 2, 15, 6.1, 1)]  # Delivery 1 + Pickup 2
522        
523        Notes:
524            - Uses Dijkstra cache via get_dijkstra_cached for performance
525            - Does not modify node state (pure method)
526            - Result should be manually assigned to self.available_points
527            - Capacity check is pre-emptive (before route calculation)
528            - Fuel check is post-route calculation (after knowing actual distance)
529        
530        Constraint Validation:
531            Capacity: Prevents overloading by checking total load after pickup

713def calculate_heuristic(state,end_points_reached,initial_points_reached,average_cost_per_task,lambda_penalty:int=2):
714    """Calculates heuristic h(n) for estimating remaining cost to goal.
715    
716    Implements an admissible heuristic based on:
717        1. Average cost per task × remaining tasks (optimistic estimate)
718        2. Penalty for active tasks (encourages completing pickups before starting new ones)
719    
720    This heuristic function is critical for A* efficiency. An admissible (never overestimating)
721    heuristic guarantees optimal solution. This function does not involve FIPA protocol.
722    
723    Formula:
724        h(n) = average_cost_per_task × (total_tasks - completed_tasks)
725               - λ × active_tasks
726    
727    Where:
728        - total_tasks = len(state)
729        - completed_tasks = len(end_points_reached)
730        - active_tasks = len(initial_points_reached) - completed_tasks
731        - λ = lambda_penalty (default: 2)
732    
733    Admissibility:
734        The heuristic is admissible if average_cost_per_task ≤ actual minimum cost
735        per task. Since average_cost is calculated from real tasks, this is typically
736        true (may underestimate due to not considering backtracking).
737    
738    Args:
739        state (list[tuple]): Task list as [(sender, receiver, qty, orderid), ...].
740        end_points_reached (list[tuple]): Completed deliveries [(orderid, location), ...].
741        initial_points_reached (list[tuple]): Performed pickups [(orderid, location), ...].
742        average_cost_per_task (float): Average cost per task calculated at start.
743        lambda_penalty (int, optional): Penalty factor for active tasks. Defaults to 2.
744    
745    Returns:
746        float: Heuristic value h(n).
747    
748    Examples:
749        >>> state = [(3, 7, 10, 1), (5, 9, 15, 2), (2, 8, 5, 3)]
750        >>> h = calculate_heuristic(
751        ...     state=state,
752        ...     end_points_reached=[(1, 7)],  # Task 1 completed
753        ...     initial_points_reached=[(1, 3), (2, 5)],  # Tasks 1 and 2 started
754        ...     average_cost_per_task=10.0,
755        ...     lambda_penalty=2
756        ... )
757        >>> # h = 10.0 * (3 - 1) - 2 * (2 - 1) = 20.0 - 2.0 = 18.0
758        >>> print(h)  # 18.0
759    
760    Notes:
761        - Lambda penalty λ encourages vehicle to complete deliveries before new pickups
762        - Higher λ value favors routes with less concurrent load
763        - λ=0 ignores penalty (pure task count heuristic)
764        - Penalty is negative (reduces h), making nodes with active tasks more attractive
765    
766    Heuristic Properties:
767        - Admissible: Never overestimates (if average_cost ≤ minimum actual cost)
768        - Consistent: h(n) ≤ cost(n,n') + h(n') for all neighbors n'
769        - Informative: Provides meaningful guidance (better than h=0)
770    
771    Design Rationale:
772        The lambda penalty term biases the search toward completing active orders
773        rather than accumulating many concurrent pickups, which reduces vehicle
774        load and improves practical efficiency.
775    """
776    total_tasks = len(state)
777    completed_tasks = len(end_points_reached)
778    active_tasks = len(initial_points_reached) - completed_tasks
779    average_cost_per_task = average_cost_per_task
780    return (average_cost_per_task * (total_tasks - completed_tasks)) - (lambda_penalty * active_tasks)

783def A_star_task_algorithm(graph: Graph, start:int, tasks:list["Order"],capacity:int, max_fuel: int):
784    """Executes the A* algorithm to find optimal sequence of pickups and deliveries.
785    
786    Solves the Vehicle Routing Problem (VRP) with capacity and fuel constraints,
787    finding the route that minimizes total time to execute all tasks.
788    
789    This algorithm operates as an internal planning component and does not directly
790    use FIPA protocol. However, in a multi-agent system, the resulting route plan
791    may be communicated between agents using FIPA-ACL messages:
792        - Vehicle agent may REQUEST route from planner agent
793        - Planner responds with INFORM containing optimal path
794        - Vehicle may CONFIRM acceptance or REFUSE if infeasible
795    
796    Algorithm Steps:
797        1. Initialization:
798           - Clear Dijkstra cache
799           - Calculate average cost per task for heuristic
800           - Create initial state as list of (sender, receiver, qty, orderid)
801           - Create root node at location=start
802        
803        2. A* Search:
804           - Use PriorityQueue to expand nodes with lowest f(n)
805           - For each node: evaluate available points, create children
806           - Add children to queue
807           - Stop when depth = 2 * num_tasks (all tasks completed)
808        
809        3. Path Reconstruction:
810           - Traverse parent links from goal node to root
811           - Build list of (location, order_id)
812           - Reverse list to chronological order
813    
814    Args:
815        graph (Graph): Graph instance with network topology.
816        start (int): Initial vehicle node ID.
817        tasks (list[Order]): List of Order objects to be executed.
818        capacity (int): Maximum vehicle load capacity.
819        max_fuel (int): Maximum fuel tank capacity.
820    
821    Returns:
822        tuple: A tuple (path, total_time, tree) where:
823            - path (list[tuple]): Sequence of (node_id, order_id) representing route.
824                - First element: (start, None) - initial position
825                - Following elements: (location, orderid) - pickups and deliveries
826            - total_time (float): Total time to complete all tasks.
827            - tree (TreeNode): Root of search tree (for visualization).
828    
829    Examples:
830        >>> from world.graph import Graph
831        >>> graph = Graph()
832        >>> orders = [
833        ...     Order(product="A", quantity=10, orderid=1, 
834        ...           sender="w1", receiver="s1", 
835        ...           sender_location=3, receiver_location=7),
836        ...     Order(product="B", quantity=15, orderid=2,
837        ...           sender="w2", receiver="s2",
838        ...           sender_location=5, receiver_location=9)
839        ... ]
840        >>> path, time, tree = A_star_task_algorithm(
841        ...     graph=graph,
842        ...     start=0,
843        ...     tasks=orders,
844        ...     capacity=50,
845        ...     max_fuel=100
846        ... )
847        >>> print(path)
848        # [(0, None), (3, 1), (7, 1), (5, 2), (9, 2)]
849        >>> print(f"Total time: {time}s")
850        # Total time: 45.3s
851        >>> tree.plot_tree("route_search.png")
852    
853    Edge Cases:
854        - If tasks empty: returns ([(start, None)], 0.0, root)
855        - If no feasible solution: returns (None, float('inf'), root)
856            and generates visualization showing search tree
857    
858    Notes:
859        - Goal is depth = 2 * len(tasks) (1 pickup + 1 delivery per task)
860        - PriorityQueue uses (f, id(node), node) for tie-breaking by ID
861        - order_id=None in root node (initial position without associated task)
862        - Search is optimal (finds minimum cost solution) if heuristic is admissible
863    
864    Complexity:
865        - Time: O(b^d) where b is branching factor, d is depth (2*num_tasks)
866        - Space: O(b^d) for storing search tree
867        - Typical branching: 2*num_tasks initially, decreasing as tasks complete
868    
869    Integration with SPADE Agents:
870        While this function itself doesn't use FIPA, it integrates with SPADE agents:
871        1. Vehicle agent calls this function to compute optimal route
872        2. Result may be sent to other agents via FIPA INFORM message
873        3. Coordinator agent may REQUEST route updates during execution
874        4. Warehouse agents receive INFORM about pickup times from route
875    """
876    # Simplified A* algorithm implementation for task ordering
877    from queue import PriorityQueue
878    
879    # Clear Dijkstra cache for new execution
880    clear_dijkstra_cache()
881    # Calculate average cost per task
882    total_time = sum(order.deliver_time for order in tasks)
883    average_cost_per_task = total_time / len(tasks) if tasks else 0
884    # Create initial state (sender_location, receiver_location, quantity, orderid)
885    initial_state = [
886        (order.sender_location, order.receiver_location, order.quantity, order.orderid)
887        for order in tasks
888    ]
889    
890    # Create root node
891    root = TreeNode(
892        location=start,
893        state=initial_state,
894        max_quantity=capacity,
895        max_fuel=max_fuel,
896        parent=None,
897        depth=0,
898        initial_points_reached=[],
899        end_points_reached=[],
900        h=calculate_heuristic(initial_state, [], [], average_cost_per_task),
901        g=0,
902        average_cost_per_task=average_cost_per_task
903    )
904    
905    # Priority queue for A*
906    open_list = PriorityQueue()
907    open_list.put((root.f, id(root), root))
908    target_depth = 2 * len(tasks)
909    
910    while not open_list.empty():
911        _, _, current_node = open_list.get()
912        
913        # Check if we reached the goal
914        if current_node.depth == target_depth:
915            # Reconstruct path as list of tuples (location, order_id)
916            path = []
917            node = current_node
918            while node is not None:
919                if node.order_id is not None:  # Skip root node which has no order_id
920                    path.append((node.location, node.order_id))
921                elif node.parent is None:  # Root node - add only initial location
922                    path.append((node.location, None))
923                node = node.parent
924            path.reverse()
925            
926            # Return: path with tuples (location, order_id), total time, search tree
927            total_time = current_node.g
928            return path, total_time, root
929        
930        # Evaluate available points
931        current_node.available_points = current_node.evaluate_available_points(graph)
932        # Create children
933        current_node.create_childs()
934        
935        # Add children to priority queue
936        for child in current_node.children:
937            open_list.put((child.f, id(child), child))
938    
939    root.plot_tree("route_search.png")
940    
941    return None, float('inf'), root

476    def evaluate_available_points(self,graph: Graph):
477        """Evaluates which points (pickups/deliveries) are reachable from current node.
478        
479        Checks all tasks in state and determines which transitions are viable
480        considering:
481            - Capacity constraints (not exceeding max_quantity)
482            - Fuel constraints (not exceeding max_fuel)
483            - Logical sequence (pickup before delivery)
484            - Already visited points (avoid duplicates)
485        
486        This method implements the constraint checking logic that ensures generated
487        routes are feasible with respect to vehicle limitations.
488        
489        Decision Logic:
490            For each task (sender, receiver, qty, orderid) in state:
491            
492            1. If pickup not yet performed:
493               - Check if qty + current_load ≤ max_quantity
494               - Calculate route and fuel using get_dijkstra_cached
495               - If fuel ≤ max_fuel: add sender to available points (type=1)
496            
497            2. If pickup already performed but delivery not:
498               - Calculate route and fuel to receiver
499               - If fuel ≤ max_fuel: add receiver to available points (type=0)
500        
501        Args:
502            graph (Graph): Graph instance to calculate routes via Dijkstra.
503        
504        Returns:
505            list[tuple]: List of tuples (location, orderid, quantity, time, type) where:
506                - location (int): Destination node ID
507                - orderid (int): Associated order ID
508                - quantity (int): Quantity to load/unload
509                - time (float): Travel time to location in seconds
510                - type (int): 1=warehouse (pickup), 0=customer (delivery)
511        
512        Examples:
513            >>> node = TreeNode(location=0, state=[(3, 7, 10, 1), (5, 9, 15, 2)], ...)

515            >>> print(available)
516            # [(3, 1, 10, 5.5, 1), (5, 2, 15, 8.2, 1)]  # Two pickups available
517            
518            >>> # After picking up order 1:
519            >>> node2 = TreeNode(location=3, initial_points_reached=[(1, 3)], ...)
520            >>> available2 = node2.evaluate_available_points(graph)
521            >>> print(available2)
522            # [(7, 1, 10, 3.2, 0), (5, 2, 15, 6.1, 1)]  # Delivery 1 + Pickup 2
523        
524        Notes:
525            - Uses Dijkstra cache via get_dijkstra_cached for performance
526            - Does not modify node state (pure method)
527            - Result should be manually assigned to self.available_points
528            - Capacity check is pre-emptive (before route calculation)
529            - Fuel check is post-route calculation (after knowing actual distance)
530        
531        Constraint Validation:
532            Capacity: Prevents overloading by checking total load after pickup
533            Fuel: Ensures vehicle can reach destination with current tank
534            Precedence: Enforces pickup before delivery for same order
535            Uniqueness: Prevents revisiting same pickup/delivery location
536        """
537        available_points = []
538        for sender_location, receiver_location, quantity,orderid in self.state:
539            if quantity + self.quantity > self.max_quantity:
540                continue
541            if (orderid,sender_location) not in self.initial_points_reached:
542                _ ,fuel,time = get_dijkstra_cached(graph, self.location, sender_location)
543                if fuel <= self.max_fuel:
544                    available_points.append((sender_location, orderid, quantity,time, 1))
545            else: 
546                if (orderid,receiver_location) not in self.end_points_reached:
547                    _ ,fuel,time = get_dijkstra_cached(graph, self.location, receiver_location)
548                    if fuel <= self.max_fuel:
549                        available_points.append((receiver_location, orderid, quantity,time, 0))
550
551        return available_points

716

722    heuristic guarantees optimal solution. This function does not involve FIPA protocol.
723    
724    Formula:
725        h(n) = average_cost_per_task × (total_tasks - completed_tasks)
726               - λ × active_tasks
727

729        - total_tasks = len(state)

745

748

  • Lambda penalty λ encourages vehicle to complete deliveries before new pickups
  • Higher λ value favors routes with less concurrent load
  • λ=0 ignores penalty (pure task count heuristic)
  • Penalty is negative (reduces h), making nodes with active tasks more attractive
Heuristic Properties:
  • Admissible: Never overestimates (if average_cost ≤ minimum actual cost)
  • Consistent: h(n) ≤ cost(n,n') + h(n') for all neighbors n'
  • Informative: Provides meaningful guidance (better than h=0)
Design Rationale:

The lambda penalty term biases the search toward completing active orders rather than accumulating many concurrent pickups, which reduces vehicle load and improves practical efficiency.
